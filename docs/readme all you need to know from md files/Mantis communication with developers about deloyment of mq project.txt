>How are rooms created in the system? Which service handles room creation?

Rooms are created automatically as needed. If there are not enough rooms, we create new rooms.

The initial number of rooms is set in RoomTemplate.initialRooms.

In addition, the number of rooms can be limited in the RoomTemplate - maxRoom.

Rooms are created by services inherited from AbstractRoomInfoService, these are:

MultiNodeRoomInfoService - for Crash game

SingleNodeRoomInfoService - all MQ action games

BGPrivateRoomInfoService - for MQ BTG action games

see method createForTemplate()

>How are players assigned to specific rooms? Which service handles it?

First, based on the game, mode (real, free, frb - see MoneyType), bank, currency, bet, a list of rooms is selected that meets the specified criteria - see

AbstractRoomInfoService.getBestRoomInfo()

RoomTemplate template = roomTemplateService.getMostSuitable(bankId, stake, moneyType, gameType);

Collection\<RI> roomInfos = getRooms(bankId, (RT) template, stake, currency);

Next, the most suitable room is selected from this list. Among the selection criteria, the main one is the number of players in the room

see AbstractRoomInfoService.tryFindThisServerRoomAndNotFull() implementations

>What are the criteria or parameters used for room selection and matching players together?

Current number of players in the room

Maximum number of players in a room

Server ID where the room is located (excluding multinode rooms)

>Which service handle situations where there are not enough players to fill a room or too many players for a single room?

services inherited from AbstractRoomInfoService, these are:

MultiNodeRoomInfoService - for Crash game

SingleNodeRoomInfoService - all MQ action games

BGPrivateRoomInfoService - for MQ BTG action games

see tryFindThisServerRoomAndNotFull() implementations

>Is there a maximum duration for a player to remain in a room before being matched with others?

No such period exists. We do not have a mechanism for redistributing players to rooms. If the player entered the room, he will be in it until he leaves it and tries to open a new room. But there is an exception to this rule, this is the battle mode, in which you need at least two players to start the game. When a player enters such a room, a timer is turned on and if after the specified time there are no more players, then we offer the player to wait or close the room

>What happens when a player disconnects or leaves a room? How is the room updated and managed in such cases?

It all depends on what state the game is in. If the game has started, then when disconnected, we keep the player in the room until the end of the round. If the player reconnects, he will continue the game in the same room. If the player deliberately closed the game, then we sitOut player from room immediately. But there are exceptions to this rule for some types of games, we can sitOut player only after the end of the round

>How does the system handle latency or connection issues between players in the same room?

Games are designed and written in such a way that delays and problems with the connection of one player do not affect other players in any way. Freezes and lags will only be experienced by this player

>Are there any logging or monitoring mechanisms in place to track room activities, player interactions, or matchmaking statistics?

We have room logs, we have statistics.log and we have the ability to view history. Sorry, but that's all. This task has never been a priority

There are two keyspaces in GS:RCasinoSCKS(SCClusterConfig.xml) and RCasinoKS(ClusterConfig.xml). What is a main difference and purpose of these keyspaces?
SCKS - it's Strong Consistency KeySpace, is used for store frequently changed data, that is not so big, but require high level of consistency. If we change data on the one node, we need to be sure that this changes immediate visible on other node(s)
enough level ONE and TWO = enough consistent level ONE and TWO
KS - it's KeySpace, is used for store archive data, that are not changed. They don't require high level of consistency (it's enough level ONE and TWO).

There could be one more keyspace RBigStorageCasinoKS(BigStorageClusterConfig.xml), what is the purpose?
This KeySpace is used for store game history, that keep very big space.
We had to allocate it due to technical restrictions of available space of one of our clusters. It's not used for MQ server.

If I change a server config xml, how to change the DB value accordingly?
It will be updated automatically, but with some restrictions. For instance:
Create table - yes
Remove table - no
Create index - yes
Drop index - yes
Add column - yes
Remove column - no
Change column type - no
Some restrictions are due to the Cassandra, e.g.: primary key cannot be changed after creation.

Explain Thrift communication between GS and MP
MQ server doesn't keep players, PlayerSession, GameSessions, player's history, GameSettings on their own. These data are kepst on GS server.
Thrift is used to load these data from GS to MQ, and is used to send updates (to save PlayerSession, GameSessions, player's history) from MQ to GS.

Explain static games applications located in /www/html/repo/html5pc/actiongames/ folder, how these interact with GS, MP, others
