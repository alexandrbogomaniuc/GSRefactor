# Protocol: abs.gs.v1

## Overview
The `abs.gs.v1` protocol is the canonical WebSocket communication standard for slot game clients. It guarantees financial correctness through strict operation tracking, idempotent retries, and comprehensive state recovery.

## Message Envelope
All messages sent between the client and server MUST adhere to the following JSON envelope wrapper:

```typescript
interface GSMessageEnvelope<T> {
  // Protocol version, exactly "v1"
  version: "v1";
  // Unix timestamp of message creation
  timestamp: number;
  // Payload body type (e.g., 'SpinRequest', 'EnterResponse')
  type: string;
  // Unique idempotency key (generated by client for requests)
  operationId?: string;
  // The actual transaction payload
  data: T;
}
```

## Standard Message Types
### 1. Enter (Initialization)
- **Request:** Triggers session start.
- **Response:** Returns `gameState`, `bank`, `limits`, and `lastAction` (if recovering from disconnect).

### 2. Action (Game Turns)
- **Request:** E.g., placing a bet, triggering a feature, settling a round.
- **Response:** Contains `result` arrays, the updated `gameState`, and modified `bank` balances.

### 3. Error
- Standardized error codes for insufficient funds, invalid state, or timeout.

## Idempotency Rules
For every monetary transaction:
1. The client **MUST** generate a unique UUIDv4 on the first attempt and attach it as `operationId`.
2. If the connection drops or a timeout occurs, the client **MUST NOT** generate a new ID. The retry payload must reuse the *exact same* `operationId`.
3. The server ensures that executing the same `operationId` twice will yield the cached result instead of deducting funds twice.
